# 为什么需要这些文件

本仓库中的每一条规则，都对应 AI 辅助软件开发中反复出现的一种失败模式。本文说明每个文件防止什么失败，以及观察来自何处。

---

## CLAUDE.md

**来源：** Andrej Karpathy 2026 年 1 月 26 日在 X 上的帖子，描述从 80% 手写代码转向 80% Agent 驱动编码后的失败模式。开发者 Forrest Chang 将其提炼为 `andrej-karpathy-skills` GitHub 仓库。本仓库版本从 Chang 的 184 行扩展版精简为 90 行（约减少 63% token），同时保留全部行为规则。

**Karpathy 实际观察到的问题：**

> "模型会替你做出错误假设并一路执行到底，不做核对。它们不管理自己的困惑，不寻求澄清，不暴露矛盾，不呈现权衡，在该反对时不反对。"

> "它们仍然喜欢把代码和 API 过度复杂化，膨胀抽象，不清理死代码……明明 100 行够用却写出上千行的臃肿实现。"

> "它们有时仍会改动或删除自己并未充分理解的注释和代码，即便与当前任务无关。"

**规则 → 防止的失败：**

| 规则 | 防止的失败 |
|------|-----------|
| §1 先读再写 | Agent 从训练数据模式匹配而非阅读你的代码库，产出与项目格格不入的代码 |
| §2 先思考再写代码 | Agent 静默选择一种解释（「我假设用 JWT 认证」）并在错误假设上搭建 200 行代码 |
| §3 简单 | Agent 在你只需要 `send_welcome_email()` 时写出带策略模式的 `EmailService` |
| §4 外科手术式修改 | Agent 重排格式、调整 import、改变量名——200 行 diff 里藏着真正的 3 行改动 |
| §5 验证 | Agent 未运行本可发现问题的测试就宣布修复完成 |
| §6 目标驱动执行 | Agent 未说明「完成」长什么样就开始实现，代码表面过关却在关键时刻失败 |
| §7 调试 | Agent 看到 `TypeError` 就按错误类型生成修复，不读完整堆栈 |
| §8 依赖 | Agent 静默往 package.json 加三个包，而标准库已能解决问题 |
| §9 常见失败模式 | 为 Kitchen Sink、失控重构、知识幻觉等命名，便于快速识别具体失败行为 |

## Loop Engineering 技能

**来源：** *Loop Engineering: The Anthropic Playbook* — IEEE 风格工作笔记（HuaShu，Orange Books，v260615，2026 年 6 月），综合了 Addy Osmani 的「Loop Engineering」博文（2026 年 6 月 7 日）、Boris Cherny（Anthropic，Claude Code 负责人）关于写循环而非写 prompt 的公开言论、Peter Steinberger 关于设计循环来 prompt Agent 的 viral 帖子，以及 Prithvi Rajasekaran（Anthropic）关于生成器/评估器模式的研究。

**论文诊断的核心问题：**

让 Agent 循环强大的同一套工具，也让其失败变得静默且不断累积。与 prompt 层失败（一次错误答案、立刻可见）或 harness 层失败（一次糟糕运行、diff 里可见）不同，loop 层失败会在多轮之后才被人察觉。论文命名了四种静默成本——验证债务、理解腐化、认知投降、token 爆炸——且它们相互强化。

**五步法 → 跳过某步的后果：**

| 步骤 | 跳过时会发生什么 |
|------|----------------|
| Discovery（发现） | 人仍每天早上决定循环处理什么——最耗时的部分没有自动化 |
| Handoff（交接） | 并行 Agent 写同一工作目录；编辑冲突；合并无法理清 |
| Verification（验证） | 循环以机器速度生成自批准输出；从未出现 REJECT 的统计不可能被注意到 |
| Persistence（持久化） | 循环每轮重新发现相同工作；更糟的是重做并与上次尝试冲突 |
| Scheduling（调度） | 循环只在人记得跑的时候跑；上次运行还是演示那天 |

**为什么评估器有独立模板：**

论文最有实证支撑的主张是：让生成器自我批判会失败，而调优独立的怀疑型评估器会成功。Anthropic 工程师 Prithvi Rajasekaran 在长期运行的 Agent 应用中发现：编写代码时的上下文充满「为何这样写」的理由，生成器无法清晰审视自己的输出。一个被指示默认持怀疑态度的新 Agent 没有这个问题。

评估器模板比其他模板更规定性，因为它防止的失败——点头循环（Nodding Loop）——最常见也最危险。从未产出过 REJECT 的循环不是正常工作的循环。

## 模板

模板是 canonical 循环产物。loop-engineering 技能在 `skills/loop-engineering/templates/` 下内置同一套文件——BUILD 模式从那里复制，根据访谈定制 `CUSTOMIZE` 标记后写入项目。Path A（技能）与 Path B（直接从 `templates/` 复制）产出一致。

它们也放在仓库根目录 `templates/` 下，方便已明确需求的开发者直接复制，无需交互式搭建。

每个模板在需按项目调整的决策点有 `<!-- CUSTOMIZE -->` 标记。标记之外的内容是有意为之的意见，除非有充分理由否则应保持不变。

loop-triage 模板中的 `Stop` 部分绝不能删除。它编码了循环可自主做什么、什么必须有人介入的边界。IEEE 论文称之为「敞开的门」——让人能够对一台被设计成高速说「是」的机器说「不」的检查点。
